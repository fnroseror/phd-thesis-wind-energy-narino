\# Appendix H. Computational reproducibility support



\## H.1. Purpose of this appendix



This appendix documents the computational reproducibility support structure associated with the doctoral thesis repository.



Its purpose is not to claim full one-click universal execution for every historical script version, but to preserve the technical logic, execution order, software context, and traceability conditions required to understand and evaluate the computational workflow.



\## H.2. Reproducibility philosophy



The repository is organized as a scientific support environment for the doctoral thesis.  

Therefore, reproducibility is treated here in a structured and realistic sense, based on the following principles:



1\. preservation of the workflow logic,

2\. explicit documentation of data and metadata structure,

3\. separation between code, results, figures, and tables,

4\. documentation of software context,

5\. progressive improvement of portability where historical local paths may still exist.



\## H.3. Main reproducibility support files



The repository includes the following reproducibility-oriented files:



\- `09\_REPRODUCIBILITY/data\_contract.md`

\- `09\_REPRODUCIBILITY/execution\_pipeline.md`

\- `09\_REPRODUCIBILITY/software\_versions.md`

\- `09\_REPRODUCIBILITY/sessionInfo.txt`

\- `09\_REPRODUCIBILITY/validation\_checks.md`

\- `09\_REPRODUCIBILITY/repository\_version\_support.md`



Together, these files provide a documented base for technical interpretation and evaluation of the computational support environment.



\## H.4. Workflow interpretation



The computational workflow of the repository should be interpreted through the following general sequence:



1\. metadata and structural understanding of the dataset,

2\. preprocessing and traceability preparation,

3\. physical characterization of the system,

4\. predictive modeling by methodological families,

5\. uncertainty-aware interpretation,

6\. structural outputs such as FNRR and usable energy,

7\. organization of final figures, tables, and appendix-level support material.



This means that the repository is designed to preserve the logic of the doctoral computational process rather than only isolated scripts.



\## H.5. Portability note



Some scripts may preserve traces of historical local implementation paths used during the active development stage of the thesis.



These paths do not invalidate the scientific value of the repository, but they should be progressively replaced, when necessary, by more portable configurations such as:

\- relative paths,

\- configurable input/output definitions,

\- execution notes associated with local reconstruction.



\## H.6. Reproducibility interpretation limits



This appendix does not state that every single component of the repository is frozen as a containerized production environment.  

Instead, it states that the repository preserves the scientific traceability and computational logic required for academic evaluation, interpretation, and future refinement.



Therefore, the repository should be read as:

\- a doctoral scientific support artifact,

\- a reproducibility-oriented technical companion,

\- a controlled-evolution research environment.



\## H.7. Final note



The reproducibility support preserved in this repository is sufficient to sustain scientific reading, methodological interpretation, and progressive technical refinement.  

Its main value lies in maintaining coherence between the doctoral thesis, the code structure, the documented outputs, and the derivative research products.
